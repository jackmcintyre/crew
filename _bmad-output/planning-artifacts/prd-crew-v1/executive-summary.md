# Executive Summary

AI Engineering Team v1 is a locally-installable Claude Code plugin that lets a non-engineer plan, ship, review, and learn from software with the rigour a traditional scrum team enforces — but with AI agents doing the hands-on-keyboards work. The product is not an "AI coding tool." It is the *scaffolding around* AI coding work that turns generation into engineering: a project-shaped team of long-lived agents, a primed continuous-flow backlog, a written standard each PR is judged against, and retros that sharpen the standard — and the team — over time.

**The team forms itself around the project.** The first interaction is with a hiring manager agent who understands the project at a high level and proposes the starting team — dev, reviewer, planner, retro analyst, and any specialists the project warrants (security, docs, test, etc.). The user approves the hires; each hired agent gets a long-lived persona file that accretes project-specific knowledge cycle over cycle. Agents are protective of their domain — they insist on their judgement inside it — and deferential outside it, yielding to whoever the expert is via an explicit handoff protocol. v1 ships a catalogue of pre-defined specialist roles the hiring manager picks from; generative role creation (the hiring manager writing a brand-new agent spec on the fly) is a Growth-phase ambition.

The target user is a relatively technical non-engineer: someone who reads code well enough to follow what an agent did and judge whether it makes structural sense, but cannot debug, refactor, or write production code by hand. Jack is the canonical v1 user; one external reader of his eventual writeup is the v1 success test.

The problem the product solves is that AI agents shipping unreviewed code is expensive chaos. Generation is no longer the bottleneck; *trustworthy* output is. Without engineering rigour — definition of done, review against a standard, calibration when the standard misses, specialists who own their domain — a non-engineer who delegates to agents ships either nothing usable or, worse, something that works but is built on a foundation no one can maintain. The product fights both failure modes by keeping the human firmly in the product loop (what to build, what counts as done, whether it actually serves a user) while delegating the engineering loop (write, review, integrate, retro) to a team whose composition fits the project and whose personas compound knowledge over time.

## What Makes This Special

**A team that forms itself, with personas that compound.** Most AI tooling presents a fixed agent or a small set of subagents. Here, the *first interaction* is with a hiring manager who proposes a starting team shaped by the project — and once hired, every agent has a persistent persona file it reads at session start and updates at session end. Cycle over cycle, the security specialist remembers what burned the project last time. The reviewer remembers which standards have produced false positives. The planner remembers which kinds of stories tend to spawn rework. The team accretes the institutional memory a real engineering team would; the product just makes that memory observable and editable.

**Stay-in-your-lane is the team protocol, not a side rule.** Each agent has an explicit `domain:` field naming what it owns. Inside its domain, it insists on its judgement; outside it, it yields to the expert via a locked handoff phrase. This is how the team avoids becoming a committee of hedging generalists — and how rubber-stamping ("AI reviews AI and they agree") gets neutralised: a security specialist with a *protected* domain will not let a generalist reviewer wave through a missing auth check.

**The rigour is the product, not the code.** Most "AI builds software for you" tooling treats review, retros, and standards as optional polish. Here they are load-bearing: a PR cannot merge without the right agent reviewer judging it against `docs/standards.md`; a retro doesn't just record what happened, it produces structured rule and skill proposals — and over time, *team changes* (hire a debugger specialist; consider unhiring the docs specialist who hasn't fired in six cycles) — that update the standard and the team for the next cycle. The agents get sharper because the standard does. The team gets fitter because the retro tunes it.

**Continuous flow, not sprints.** Sprints were inherited from Jack's scrum-master instincts, but agent throughput is high enough that time-boxing creates friction without predictability. The product's natural shape is a primed `to-do/` queue that agents drain continuously. The user's job is keeping the queue good; the agents' job is steadily emptying it. Stories move atomically between filesystem-tracked states (`to-do/` → `in-progress/` → `blocked/`/`done/`) so multiple agent sessions can coordinate without race conditions.

**Risk-tiered autonomy, not all-or-nothing.** Low-risk PRs auto-merge once the agent reviewer signs off; medium-risk pause for the human; high-risk pause with extra scrutiny. The tier definitions are explicit (path-based, change-type-based, diff-size-based) and evolve from observed misses, not vibes. The user trades a small, controllable risk surface for a large reduction in babysitting load.

**The calibration loop is the moat.** A traditional team's quality compounds through institutional memory: standards docs, code review culture, retros, postmortems. The product replicates this loop mechanically: every PR is a paired observation (agent verdict + human merge decision), every divergence is a retro signal, every retro proposes a rule or skill update, every accepted proposal sharpens the standard the agents work against. The product designs for its own improvement; without that, AI generation drifts toward mediocrity instead of away from it.

**The differentiator moment.** Two of them, in sequence. First: the user installs, opens a hiring conversation, and watches the hiring manager propose a team shaped by *their* project — not a generic roster — with each role justified by what's in the repo. Second: the user closes the laptop with a primed backlog and opens it later to find reviewed, merged PRs, a retro file naming what the agents learned, *and* persona files that show what each agent now knows about this project that it didn't yesterday. Those two moments together are what makes the product something different from "an AI that writes code." It is a team that formed around the work, did the work while the user was away, and got smarter in the process.

## Core Insight

"Shipped well, used by no one" — Jack's stated worst case — happens when engineering rigour is decoupled from product judgement. The product enforces the split deliberately: the human owns the *product* loop (vision, prioritisation, what counts as done at the user-value level, whether to keep building this at all); the agents own the *engineering* loop (writing, reviewing, integrating, retrospecting). The backlog is the contract between them. If the backlog is right, the agents repeatedly ship the right thing without supervision. If the backlog is wrong, the product fails fast — because a non-engineer cannot rescue a misdirected sprint by hand, they can only re-prime the queue. This is a feature, not a bug: it forces the user's attention onto the question that actually determines whether the software gets used.
