# Implementation Patterns & Consistency Rules

## Conflict Points Identified

The default pattern categories (REST endpoint naming, JSON wrappers, DB naming) do not apply to this project. The real conflict surfaces for AI Engineering Team v1 are the artifacts agents author and the wire formats between sessions:

1. Markdown frontmatter conventions (story, persona, catalogue, retro proposal)
2. Story file body shape (sections, AC formatting)
3. Catalogue & persona file shape
4. MCP tool naming
5. JSONL event schema
6. TypeScript code conventions (file naming, imports, errors)
7. Locked phrases (handoff, verdict, yield)
8. Skill file shape (slash-command Markdown)
9. Telemetry field conventions
10. Commit & PR conventions for the dev subagent
11. `gh` allowlist file format
12. Risk-tier classifier output shape

## 1. Frontmatter Conventions (all *plugin-owned* Markdown / YAML artifacts)

These conventions apply to artifacts the plugin authors: catalogue specs, persona files, execution manifests (`.crew/state/<state>/<ref>.yaml`), rule registry, retro proposals, skill files, workspace config. **Source story files are governed by their adapter's contract** (see ¬ßPlanning Adapter Model) ‚Äî BMad's frontmatter is BMad's; we read it through the adapter's normalisation, we don't enforce our conventions on it.

- **YAML only**, in the leading `---`-delimited block.
- **Key naming: `snake_case`** ‚Äî matches PRD examples (`depends_on`, `blocked_by`, `claimed_by`, `risk_tier`, `failure_class`, `rework_count`).
- **Dates: ISO-8601 strings** (`2026-05-19T14:30:00Z`). Never epoch ints.
- **IDs: ULID strings**, lower-case.
- **Enums: bare strings, lower-case**, kebab-allowed.
- **Lists: YAML block sequences** (one `- item` per line) ‚Äî not flow sequences ‚Äî so diffs stay line-oriented.
- **Validation:** every frontmatter shape has a Zod schema in `mcp-server/src/schemas/`. Malformed files raise the human-readable error required by FR13.

## 2. Story File Body Shape

**Applies to the `native` adapter only** ‚Äî i.e. users without a planning tool whose stories are authored by the plugin's own planner. External adapters (BMad, etc.) leave story body shape to the source tool; the adapter is responsible for projecting whatever the tool produces into the *normalised internal `SourceStory` shape* the rest of the plugin consumes (see ¬ßPlanning Adapter Model).

For the native adapter, sections in this order, exactly these `##` headers:

```markdown
# Narrative
<one paragraph: what the user gains>

# Acceptance Criteria
- AC1 (integration): <runnable check>
- AC2 (unit-level): <‚Ä¶>

# Implementation Notes
<optional; planner's hints; dev may rewrite>

# Dependencies
<auto-generated mirror of frontmatter `depends_on`; do not hand-edit>
```

- ACs are numbered (`AC1`, `AC2`, ...).
- Integration ACs are explicitly tagged `(integration)`. Required for state-mutating stories (FR5).
- The `Dependencies` section is regenerated by the MCP `validateStory` tool; agents must not edit it directly.

The **normalised internal `SourceStory` shape** (what adapters emit, and what the rest of the plugin consumes) is the contract that matters across adapters ‚Äî defined in ¬ßPlanning Adapter Model.

## 3. Catalogue & Persona File Shape

Both share the same `##` skeleton; the persona file adds the knowledge section.

```markdown
---
role: <kebab-case-role-id>
domain: "<short noun phrase, exact-match string>"
model_tier: <opus | sonnet | haiku>
tools_allow: [Read, Edit, Bash, ...]
gh_allow: [pr-create, pr-view, ...]
locked_phrases:
  handoff: "Handoff to <next role> ‚Äî <intent>"
  yield: "This sits in <role>'s domain ‚Äî handing off"
  verdict: "**Verdict: <SENTINEL>**"
hired_at: <ISO-8601>           # persona files only
catalogue_version: <plugin semver>  # persona files only
---

# <Display name>

# Domain
<one paragraph; expands on the frontmatter `domain:` string>

# Mandate
<bulleted list of what this role owns>

# Out of mandate
<what to yield to others on>

# Prompt
<the agent's system-prompt body, authored as Markdown>

# Knowledge   # persona files only
<empty on hire; cycle-over-cycle appends land here via accept-proposal>
```

- **`domain:` is the routing key.** Exact match required by FR99. Load-bearing ‚Äî never edit casually.
- **`tools_allow` and `gh_allow` are the permission contract.** Enforced at the MCP / execa boundary (NFR12).

## 4. MCP Tool Naming

- **Convention:** `camelCase` verb-noun (`claimStory`, `recordVerdict`, `appendPersonaKnowledge`, `computeOutcomeStats`, `lookupRoleByDomain`).
- **Flat namespace, no dot-prefixes** ‚Äî MCP clients call by single name; namespacing adds friction without payoff at v1 tool count (~20‚Äì30 tools).
- **Mutators**: name starts with a verb of mutation (`claim`, `complete`, `record`, `append`, `apply`).
- **Readers**: name starts with `get`, `list`, `lookup`, or `compute`.
- **Renames are breaking changes** ‚Äî bumped in the plugin semver.

## 5. JSONL Event Schema

```jsonc
{
  "ts": "2026-05-19T14:30:12.345Z",       // ISO-8601 with ms
  "type": "reviewer.verdict",             // discriminator (dotted)
  "session_id": "<ulid>",
  "story_id": "<ulid>",                   // optional, included when relevant
  "agent": "generalist-reviewer",         // role id
  "data": { /* type-specific payload */ }
}
```

- **One event per line, no trailing comma.** Strict JSONL.
- **`type` discriminator: `domain.event` dotted** ‚Äî `agent.invoke`, `reviewer.verdict`, `yield.handoff`, `retro.proposal`, `state.transition`, `team.change`, `persona.append`, `skill.invoke`.
- **`data` is a closed schema per `type`.** Zod-validated on write.
- **No nested unbounded objects.** If a payload grows, add a new event type ‚Äî don't generalise an old one.
- **No PII / no diff contents in telemetry.** Telemetry stays local, but the convention keeps surprises out of the file (NFR14).

## 6. TypeScript Code Conventions (MCP server source)

- **File naming: `kebab-case.ts`** (`claim-story.ts`, `verdict-grammar.ts`). Test files: `*.test.ts` co-located.
- **Module structure**: `src/tools/`, `src/state/`, `src/schemas/`, `src/lib/`, `src/validators/` ‚Äî matches the layout left behind by sprint-orchestrator.
- **Imports**: relative within the package; no path aliases in v1.
- **Errors**: typed `class StoryNotFoundError extends DomainError {‚Ä¶}` for each named failure mode; base `DomainError` carries a `code` string. Tool handlers catch `DomainError` and convert to MCP error responses; uncaught errors are bugs.
- **No `any`.** Schemas are the source of types via `z.infer`.
- **No default exports.** Named only.

## 7. Locked Phrases (exact strings)

| Where | Phrase |
|---|---|
| Dev ‚Üí Reviewer handoff | `Handoff to reviewer ‚Äî story <story-id> ready for review.` |
| Yield to specialist | `This sits in <role>'s domain ‚Äî handing off.` |
| Verdict line (must be final line of summary comment) | `**Verdict: READY FOR MERGE**` / `**Verdict: NEEDS CHANGES** [<N> issues, <M> questions]` / `**Verdict: BLOCKED** [<reason>]` |
| Verdict footer marker (HTML comment) | `<!-- crew:verdict:<plugin-version>:<story-id> -->` |
| Routing-failure surface | `[routing-failure] no hired role matches domain "<x>"` |

- These strings are **tested**. Changes ship as breaking plugin-semver bumps.
- Reviewer cannot stack verdicts ‚Äî re-run finds-by-footer-marker and edits in place (NFR11).

## 8. Skill File Shape (slash-command Markdown)

```markdown
---
name: <plugin>:<command>
description: <one sentence; shown in /<plugin> tab-complete>
allowed_tools: [Read, Edit, Bash(execa-allowlist), Task]
---

# /<plugin>:<command>

# What this skill does
<short prose>

# Prerequisites
<what must exist in the target repo>

# Steps
1. <‚Ä¶>
2. <‚Ä¶>

# Failure modes
<known errors and their meanings, in plain language>
```

- Hiring manager and retro analyst can *propose* new skills; they go through `accept-proposal` and write to `plugins/<plugin>/skills/` (or the user's personal skill dir for retro-proposed user-skills).

## 9. Commit & PR Conventions (dev subagent)

- **Branch**: `story/<story-id>-<slug>` (slug is kebab-case-from-title, trimmed to 40 chars).
- **Commits**: conventional-commits (`feat:`, `fix:`, `refactor:`, `test:`, `docs:`); body wraps at 72.
- **PR title**: `<conventional-type>(<story-id>): <story title>`.
- **PR body**: machine-section first (story link, ACs as a checklist mirrored from the story file), then a free-form summary.
- **No `--no-verify`, no `--force-with-lease` without user request.** Encoded in the dev role's prompt and enforced via the `gh` allowlist.

## 10. gh Allowlist File Format

`plugins/<plugin>/permissions/<role>.yaml`:

```yaml
role: generalist-dev
gh_allow:
  - pr-create
  - pr-view
  - pr-comment
  - api  # restricted by gh_allow_args below
gh_allow_args:
  api:
    - "repos/:owner/:repo/issues/:issue/comments"   # read-only paths
```

- One file per role; reviewed in PRs alongside catalogue changes.
- `execa` wrapper rejects any subcommand not in the role's `gh_allow`.

## 11. Risk-Tier Classifier Output Shape

```jsonc
{
  "story_id": "<ulid>",
  "tier": "low" | "medium" | "high",
  "matched_rule": "<rule-id from risk-tiering.md>" | "fallback",
  "evidence": { "paths": [...], "change_types": [...], "diff_size": <int> }
}
```

- Stamped into the story frontmatter (`risk_tier`) and the verdict comment.
- Fallback (no rule matched) ‚Üí `medium` + `matched_rule: "fallback"`.

## 12. Enforcement

**All AI agents MUST:**

- Author and read Markdown artifacts via the MCP server's read/write tools ‚Äî never raw `fs.write` to canonical paths.
- Use the locked phrases verbatim. Paraphrasing breaks routing.
- Validate every frontmatter shape via its Zod schema before persisting.
- Emit a JSONL event for every action that mutates canonical state.

**Pattern violations:**

- Frontmatter schema failure ‚Üí hard error to the calling agent (the MCP tool refuses the write).
- Locked-phrase drift ‚Üí caught by the verdict-grammar parser; reviewer's response is treated as `BLOCKED [reviewer-grammar-error]` and labelled `needs-human`.
- Telemetry-event schema failure ‚Üí write rejected; surfaces as a story-level retro `tool-quirk` lesson.

**Process for updating patterns:**

- Patterns are **enforced by code** (Zod schemas, grammar parsers, file-naming lints). Changes go through the same PR flow as application code.
- Any pattern change that breaks existing artifacts is a breaking plugin-semver bump.
