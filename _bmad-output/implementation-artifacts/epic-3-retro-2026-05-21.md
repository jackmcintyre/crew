## What worked

### The 3.3 → 3.3b correction loop
Story 3.2's reviewer flagged a hard-coded `if (activeAdapterName === "bmad")` branch left in `scan-sources.ts` because the workspace resolver wasn't ready yet. It was caught in 3.2's retro, scoped as a focused story (3.3b), and shipped clean (review pass 1 of 3). This is the pattern working: reviewers spot tech debt, retros propose follow-ups, follow-ups become their own small PRs. The cost of catching it in retro was one extra PR; the cost of leaving it would have grown with every new adapter.

### Variable review-pass budget
The 3/5 budget split (substrate vs user-surface) introduced in Epic 2 paid off. Substrate stories (3.1, 3.3, 3.3b) all approved on pass 1. User-surface stories ran 1–3 passes. Nothing hit the budget ceiling. Average across the epic was ~1.6 passes — below the 1.5 canary line is the goal; we are slightly above but well within the spirit.

### Spec validation caught a real contradiction (3.7)
The Step 5 spec-validation gate caught an example-vs-rule contradiction in 3.7 before any dev tokens burned — AC4's example called a four-arg function with two args. One re-author cycle fixed it. That gate exists for exactly this case and it justified itself.

### Operator-smoke caught bugs automated tests missed
- **3.4 smoke** — surfaced a scary-looking `git rev-parse failed: HEAD` from the planner against a fresh `git init` scratch repo. Wouldn't have shown up in any test.
- **3.5 smoke** — verified the planner refusal flow end-to-end with both `missing-integration-ac` and `missing-ship-gate` violations cited, no file written. The behavioural contract held.
- **3.6 smoke** — caught a backlog-display dedup bug (native ULIDs listed twice). Non-blocking but real.

When you can actually run the surface, you find things. The blocker (see below) is that running the surface was painful.

### Native adapter unified cleanly with BMad
Story 3.4's `NativeAdapter.listSourceStories()` returns objects with the same key set as BMad's. The `PlanningAdapter` interface (Story 3.1) earned its keep — the second adapter didn't need any interface changes. That's a sign the abstraction was right-sized.

## What hurt

### The dev loop — easily the biggest tax on the epic
Story 3.2 was the first user-surface story to need real-Claude-Code evidence on a slash command (`/crew:scan`). The path that *should* have worked — `pnpm dev:install` → `/plugin install` → run the slash command — didn't:

1. The marketplace source resolved to `main`, not the worktree. New files weren't visible to the installed cache.
2. Overlaying files into `~/.claude/plugins/cache/.../` looked right structurally but Claude Code's daemon caches the skill index across sessions and won't re-scan.
3. Uninstall + reinstall wiped the overlay and re-copied from main.
4. Story 1.11's symlink-based dev install (the supposed fix) got *deleted* by Claude Code's plugin healer on restart — it expects a directory-copy install, sees a symlink, treats it as corruption.

3.4's retro found the workaround: launch Claude Code with `claude --plugin-dir <worktree>/plugins/crew`. That's Anthropic's blessed dev workflow ([issues #17361](https://github.com/anthropics/claude-code/issues/17361), [#23819](https://github.com/anthropics/claude-code/issues/23819)). It bypasses the marketplace cache entirely. By 3.6, this was the working pattern; ship-story SKILL.md was updated to instruct operators correctly.

**The cost:** Story 3.2 shipped its AC4 via the "automated route" (SKILL.md content-anchor tests as structural evidence), skipping the human-eyes-on-the-TUI check the gate was designed to enforce. Defensible but not the bar we set.

**Strategic implication:** every user-surface story in Epic 4 will need operator-smoke. The `--plugin-dir` workflow is documented now but the dev-install symlink approach (Story 1.11) is still in the repo and is broken-by-design. There's a follow-up story floated in 3.4's retro: replace dev-install with a `--plugin-dir` launcher script.

### `cd`-leak in ship-story orchestration (bit twice)
Stories 3.4 and 3.6 both hit the same footgun: the Bash tool's cwd persists across calls, and an earlier subshell `cd` into the worktree (or into a subdirectory) caused subsequent `ship.py record` calls to write state into the wrong `.claude/` tree. 3.4 was the worst — an attempted recovery `rm -rf .claude` came close to clobbering the shared ship-story run-log tree. Recovered by `git checkout`, but a near-miss with real lost-work potential. Memory entry already exists about not `cd`-ing into subdirs; the gap is that it doesn't extend to `record-verification` or to ship.py self-rejecting when its REPO root disagrees with cwd. 3.4's retro proposes the fix; not yet implemented.

### Story 3.5 was the costliest review cycle of the epic
Three review passes on a single story. Pass 1 found 6 issues including one High (spec/implementation drift on the remediation flow). Pass 2 found a spec-lock-vs-new-behaviour drift that required a spec amendment mid-flight. Pass 3 approved with 7 Info-tier residual issues shipped. Two structural lessons:
- **Spec amendments are untracked.** 3.5's spec amendment lives only on local disk because `_bmad-output/implementation-artifacts/` is gitignored. Future readers and remote agents can't see what was agreed. This is a contract-history hole.
- **Spec locks need a "carve-out" path.** When the spec says "MUST NOT modify file X" but the implementation legitimately needs to touch X for the AC to be satisfied, there's no clean process for relaxing the lock. The reviewer caught it, the dev re-authored an amendment, but the loop cost a pass.

### The AC3-without-caller pattern (3.7)
Story 3.7's AC3 promised: *"the next skill invocation flags the edit and refuses to proceed."* But no v1 caller invokes the `detectInProgressHandEdit` guard. The pre-PR gate (Story 1.8) caught it correctly and asked for user-surface evidence. There was no chat surface to point at, so the verification was recorded via the predicate's unit tests — `automated_e2e_verified`, not operator-smoke. The gate accepted that, but the AC's user-surface promise is paper-only until Epic 4 wires the first caller.

**Pattern to avoid going forward:** don't ship user-surface ACs whose chat surface depends on a deferred caller. Two safer shapes:
- Ship the caller in the same story (preferred).
- Split into a substrate predicate story now and a separate user-surface wiring story later — and don't tag the predicate-only story as user-surface.

3.7 did neither honestly. Worth a one-liner in the user-surface AC rubric (`plugins/crew/docs/user-surface-acs.md`) so a future planner catches it.

## Strategic state of the product

**What a non-engineer can do today:**
- Drop the plugin into a native repo, run `/crew:plan`, talk to the planner subagent, get a backlog of stories under `.crew/native-stories/`.
- Drop the plugin into a BMad repo, run `/crew:plan`, get pointed at the right BMad authoring skills.
- Run `/crew:scan` to produce execution manifests in `.crew/state/to-do/`.
- Re-open the planner mid-cycle, discard a feature (built or pending), edit pending stories.
- Hand-edit any `to-do/` or `blocked/` manifest in their editor and have the next skill invocation honour the edit.

**What does not yet work end-to-end:**
- The dev/review loop (claim → implement → review → merge). That's Epic 4.
- The `detectInProgressHandEdit` refusal is plumbed but not wired into any caller — operator hand-edits to `in-progress/` will silently desync until Epic 4 wires the guard into the claim path.

**The "can a non-engineer prime a backlog, walk away, come back to merged PRs?" test:** primed half is done. The walk-away half is Epic 4 + 5.

## Closing note

The biggest thing Epic 3 surfaced is that **the dev loop is product code now**. Every user-surface story has to be runnable in a real Claude Code session, by a human, to validate the AC. The tooling for "spin up a scratch repo, point Claude Code at the worktree, run the slash command, paste the output back" was hand-rolled per story across 3.2 → 3.4 → 3.6 until a stable pattern emerged. Formalising that pattern (one launcher, one helper to mint scratch manifests, one record-verification flow that doesn't require an operator to remember `cd`) would pay back across all of Epic 4, 5, 6, and 7.

If there's one Epic 4 story to add ahead of the planned roster, that's it.
